# 穿剑 - 问题排查与修改清单

> 审查范围：server/src 全部逻辑 + client/js 全部脚本。按严重程度排序，每条给出现象、复现、定位、建议改法。

## P1 严重（影响公平/直接破坏对局）

1. 炸弹倍数在"出牌失败"时也会被累加
   - 现象：出一个管不上的炸弹（小炸弹压不过大炸弹）被拒绝后，最终积分倍数仍然翻倍。
   - 复现：A 出四个 K，B 选四个 7 点"出牌"被判"管不上"，但 bombCount 已 +1，整局 score x2。
   - 定位：server/src/game/gameManager.js playCards，`if (handAnalysis.type === 'bomb') this.bombCount++;` 在 canBeat 校验之前执行。
   - 改法：把 bombCount++ 移到所有校验（INVALID / canBeat）通过、且 lastPlay 真正更新之后再累加。

2. 对局中途有人断线会让整局错乱
   - 现象：真人中途断线/退出后，剩余玩家轮次混乱、可能卡死或回合指向错误的人。
   - 复现：4 人开局后任一真人断线。
   - 定位：server/src/room/roomManager.js leaveRoom 在 isPlaying 时仍 filter 掉玩家并 forEach 重排 index，导致 room.players（变 3 人、重排）与 game.players（仍 4 人、原索引）错位；scheduleBotTurn 用 room.players[game.currentTurnIndex] 取到错误/undefined。
   - 改法：游戏进行中不要删除/重排玩家。标记该座位 disconnected=true 保留索引；提供真正的断线重连（按昵称或持久 token 回填 socketId），或在进行中由 AI 接管该座位。

## P2 重要（功能缺失/规则不符/可玩性）

3. "三不带"（三张同点不带牌）无法打出
   - 现象：手里三个 7（或除 6/Q 外任意三同点）选中出牌被判"无效牌型"，但 README 把"三不带"列为基本牌型。
   - 定位：server/src/game/rules.js analyzeHand 中 n===3 只识别 44A / 666 / QQQ，没有普通三条分支，最终落到 INVALID。
   - 改法：补一个普通三条牌型（如 THREE），给定 HAND_POWER（介于对子和三带一之间），并在 canBeat 同型按点数比较；同时确认 README 与实现一致。

4. 炸弹大小用"总张数"而非"同点张数"比较
   - 现象：四张同点 + 1 张杂牌（5 张，被识别为 bomb 且 length=5）能压过干净的四张炸弹（length=4）。
   - 定位：server/src/game/rules.js analyzeHand 炸弹 length: n；canBeat 中炸弹比较用 curLen/lastLen（=总张数）。
   - 改法：要么禁止"四带一"（炸弹只允许纯 4+ 同点），要么 canBeat 改用 bombCount 比较张数。建议统一为"炸弹=纯同点"。

5. 服务器没有回合超时，超时只靠客户端
   - 现象：当前玩家若挂起页面/切后台，服务器永远等待，整局卡死（机器人只在自己回合被 scheduleBotTurn 触发）。
   - 定位：server/src/index.js 无任何 turn timer；只有 client/js/game.js startCountdown 在本地 20s 后自动过/出最小牌。
   - 改法：服务器为真人回合加权威计时器，超时自动 pass（无 lastPlay 时自动出最小合法牌），并广播，保证对局推进。

6. "在线匹配"完全失效（前端发事件、后端无处理）
   - 现象：点"在线匹配"弹出匹配中遮罩，但永远停在 1/4，不会开局。
   - 定位：client/js/app.js startOnlineMatch 发 quick_match、cancelMatch 发 cancel_match；server/src/index.js 没有这两个事件的 socket.on；roomManager.createMatchRoom 也从未被调用。
   - 改法：实现 quick_match/cancel_match 处理：维护 matchQueue，满 4 人用 createMatchRoom 建房开局；不足时定时补机器人。或先隐藏该入口避免误导。

7. 跨局累计积分永远是 0
   - 现象：座位昵称旁的 [+x] 累计分始终为空/0，多局不累计。
   - 定位：server/src/room/roomManager.js updateScores 已实现但从未被调用；game_over 后没有把 result.perPlayerScores 写回 room.scores。
   - 改法：在两处 game_over（index.js play_cards 处理 与 scheduleBotTurn 内）调用 roomManager.updateScores(roomCode, result.perPlayerScores)。

8. "队友推断 AI"（配合推断）实际未生效
   - 现象：difficulty='ai' 的协作规则（不压队友、助队友收尾、堵对手）几乎从不触发，退化为 mediumBot。
   - 定位：(1) server/src/index.js 创建了 teamTrackers 但从不调用 observeAction/updateAllTrackers，信念分恒为 0；(2) server/src/game/aiBot.js getEffectiveTeammate/isOpponent 依赖 gameState.teammateIndex，而 gameManager.js getGameState 从不暴露 teammateIndex（只给 teammateRevealed 和每人 isTeammate）。
   - 改法：每次出牌/过牌后构造 lastAction 调用 updateAllTrackers；并让 getEffectiveTeammate 用每个玩家的 isTeammate 字段识别队友（或在 state 中补 teammateIndex，仅在 teammateRevealed 时下发）。

9. 开发者模式调用了不存在的方法
   - 现象：触发 dev_mode 时抛 TypeError（被全局 uncaughtException 吞掉），回调不返回，前端卡住。
   - 定位：server/src/index.js game.startDevMode(...)，但 gameManager.js 没有 startDevMode 方法。
   - 改法：改用现有 game.start(...)，或新增 startDevMode；若没有入口则删除该 handler。

## P3 次要（健壮性/潜在崩溃/边角）

10. canBeat 的空值保护是死代码
    - 定位：server/src/game/rules.js 先 const {...} = lastPlay 解构，再 if (!lastPlay) return true。lastPlay 为 null 时解构先抛错，保护永不生效。
    - 改法：把 if (!lastPlay) return true 移到解构之前。

11. 找不到红桃 4 时的兜底会发出第 14 张牌
    - 定位：server/src/game/gameManager.js start 中 heart4Index<0 时给 player0 push 一张 H4 而不移除任何牌（满 52 张发牌理论上不会触发，但逻辑不自洽）。
    - 改法：兜底改为与某张牌交换，或直接断言 H4 必被发出。

12. scheduleBotTurn 无防重入/防叠加锁
    - 定位：server/src/index.js。每个动作都 scheduleBotTurn，多个 2s 定时器可能叠加，理论上存在机器人重复行动/抢跑的竞态。
    - 改法：每房只保留一个待执行定时器（保存 timer id，调度前 clearTimeout），或加回合序号校验。

13. showTeammateReveal 对 const 变量再赋值
    - 定位：client/js/game.js const playerIdx = data.teammateIndex; if (playerIdx === undefined) playerIdx = ...，分支命中会抛 TypeError（当前 server 总带 teammateIndex，属死分支）。
    - 改法：改成 let，或删除该兜底分支。

14. 开局后非庄家短暂看不到手牌
    - 定位：server/src/index.js start_game 给所有人发 game_state: undefined（客户端忽略），真正手牌要等庄家叫牌后广播。叫牌阶段非庄家桌面为空。
    - 改法：start_game 后直接给每位玩家发各自真实 getGameState(i)。

15. 文档/实现细节不一致（低优先）
    - README 写"52 张标准扑克（可选大小王模式）"，但 deck.js 无大小王、也无开关。
    - 结果倍数 README 称"每出一次炸弹 x2"，实现为 2^bombCount，与 #1 叠加时偏差更大。
    - 改法：对齐文档与实现，或实现缺失开关。

## 验证建议（最小用例）

- 三条：构造手牌含三个 7，断言 analyzeHand 返回三条且可作为合法首出。
- 炸弹倍数：先出大炸弹再尝试小炸弹被拒，断言 bombCount 不变。
- 炸弹比较：四带一不应压过纯四张炸弹（或被禁止）。
- 断线：开局后移除一名真人，断言剩余玩家回合不错乱。
- 累计分：连打两局，断言 room.scores 正确累加。

建议为 rules.js / gameManager.js 增加一组纯逻辑单元测试（Node 自带 node:test 即可），牌型与比较是 bug 高发区。
